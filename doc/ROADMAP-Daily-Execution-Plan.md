## برنامه روزانه/هفتگی جامع از روز ۰ تا لانچ MVP (نگاه کلان مدیریتی)

این فایل بر اساس `ROADMAP.md` و فازهای ۱ تا ۴، یک **نقشه روزانه در قالب الگوی هفتگی** برای مدیریت و هماهنگی بین تیم‌ها ارائه می‌دهد تا از **روز ۰** تا **لانچ MVP** تصویر شفاف داشته باشید.

فازها (طبق ROADMAP + Phase4):

1. فاز ۱ – تحقیق و استراتژی (تقریباً ۴ هفته)  
2. فاز ۲ – طراحی و تجربه کاربری (حدود ۸–۱۰ هفته)  
3. فاز ۳ – فنی و معماری (۴–۶ هفته، بخشی موازی با Sprint 0/1)  
4. فاز ۴ – توسعه و تست MVP (تقریباً ۲۱ هفته تا آماده شدن برای Launch/Beta)  

در این سند:

- روی **تقویم مدیریتی و Cross-team** تمرکز می‌کنیم  
- جزییات ریز هر فاز در فایل‌های فاز مربوطه آمده است؛ این‌جا آن‌ها را به هم متصل می‌کنیم  

---

## الگوی ثابت کاری برای همه فازها (در سطح مدیریت)

### شنبه
- **هم‌ترازی هفتگی (Weekly Alignment)**
  - مرور OKRها/KPIهای کلان (Product, Tech, Growth)  
  - هم‌ترازی بین Product, Tech, UX, Legal/Marketing روی اهداف هفته  
  - تأیید تسک‌های روزمره تیم‌ها (مطابق فایل‌های فاز ۱–۴)  

### یکشنبه
- **پیگیری اجرای کارهای عمیق**
  - اطمینان از این‌که تیم‌ها روی «High-impact tasks» کار می‌کنند (نه فقط ریزتسک‌ها)  
  - حذف Blockerها از طریق تصمیم‌گیری سریع مدیریتی  

### دوشنبه
- **میان‌هفته: کنترل ریسک و وضعیت**
  - بررسی ریسک‌های جدید (فنی، بیزینسی، زمانی)  
  - به‌روزرسانی Risk Register (در Phase4)  
  - تنظیم اقدامات اصلاحی سریع (اگر تاخیر یا مشکل جدی وجود دارد)  

### سه‌شنبه
- **هم‌ترازی بین فازها**
  - اطمینان از این‌که خروجی فاز قبلی به‌موقع به فاز بعدی منتقل شده (مثلاً: Research → UX → Dev)  
  - جلسات Hand-off کوچک در صورت نیاز (مثلاً UX ↔ Dev برای یک صفحه خاص)  

### چهارشنبه
- **مرور دستاوردهای هفته**
  - چه چیزهایی در این هفته Done شده؟  
  - آیا به Milestoneهای این ماه نزدیک شده‌ایم؟  
  - آماده‌سازی مواد لازم برای ریویو پنج‌شنبه/جمعه  

### پنج‌شنبه – روز گزارش و ریویو هفتگی
- **هر هفته:**  
  - دریافت گزارش کوتاه از هر تیم:
    - Research/Strategy  
    - UX/UI  
    - Tech/DevOps  
    - Development/QA  
    - (در فازهای نزدیک به Launch: Legal/Marketing)  
  - به‌روزرسانی:
    - `ROADMAP.md` (در حد Milestone/فاز)  
    - وضعیت فازهای ۱–۴ (فقط سرفصل «معیارهای موفقیت»)  

### جمعه – روز تصمیم‌گیری و تنظیم هفته بعد
- **مدیریت (Founders + PM + Tech Lead):**
  - مرور KPIهای سطح بالا (Registered Users / Velocity / Burn-down / آماده‌بودن MVP و …)  
  - تصمیم‌گیری روی:
    - تغییر Scope، جابه‌جایی Featureها بین قبل/بعد از MVP (در صورت نیاز)  
    - تأیید شروع فاز بعدی یا ادامه فاز فعلی  
  - نوشتن یک **خلاصه یک‌صفحه‌ای** از وضعیت برای خود/سرمایه‌گذاران (در صورت نیاز)

---

## فاز ۱ (هفته ۱–۴) – Focus مدیریتی

### شنبه تا چهارشنبه
- اطمینان از این‌که:
  - تحلیل رقبا، BMC، تعریف MVP، Market Size طبق برنامه پیش می‌رود  
  - تیم UX و Tech از ابتدا در بحث‌ها حضور دارند (برای جلوگیری از Gap)  

### پنج‌شنبه
- دریافت و مرور:
  - نسخه به‌روزشده **Phase1-Research-Strategy**  
  - خلاصه یک‌صفحه‌ای یافته‌ها و تصمیم‌ها  

### جمعه
- تصمیمات کلیدی:
  - تأیید رسمی MVP Scope و BMC نسخه ۱٫۰  
  - اجازه شروع فاز ۲ و ۳ بر اساس خروجی فاز ۱  

---

## فاز ۲ (تقریباً هفته ۵–۱۴) – UX / Design

در این بازه زمانی، از دید مدیریت:

- **هفته‌ای یک بار (پنج‌شنبه/جمعه):**
  - بررسی وضعیت خروجی‌های زیر:
    - User Stories → Complete & Approved  
    - Sitemap → Final  
    - Wireframes → نسخه ۱٫۰ برای Flows اصلی  
    - Mockups High-fi → مطابق MVP  
    - Design System → نسخه ۱٫۰  
    - Prototype + Usability Test Report  

### شنبه–چهارشنبه (هر هفته)
- تنظیم اهداف هفتگی UX:
  - هفته‌های اول: Wireframes  
  - بعدی: Mockups + Design System  
  - بعدی: Prototype + Testing  
- پیگیری این‌که:
  - تسک‌های تعریف‌شده در `Phase2-Daily-Execution-Plan` جلو می‌روند  
  - تداخل/ابهامی بین UX و Tech باقی نماند

### پنج‌شنبه
- دریافت Demo سریع از UX:
  - Prototype جدید  
  - صفحات کلیدی که آماده تحویل به Dev هستند  

### جمعه
- تصمیم‌گیری روی:
  - Design Freezeهای مرحله‌ای (برای اجازه‌دادن به Dev که شروع کند)  
  - انتقال رسمی صفحات مشخص به Backlog توسعه (فاز ۴)  

---

## فاز ۳ (تقریباً هفته ۸–۱۴، بخشی موازی با فاز ۲) – Architecture

مدیریت باید اطمینان حاصل کند که:

- **Stack و معماری تا قبل از شروع Sprint 1 فاز ۴ منجمد شده است.**  
- **DB Schema، API Contract، Security Baseline، DevOps Setup** آماده و مستند شده است.  

### شنبه–چهارشنبه
- پیگیری از Tech Lead:
  - آیا بخش‌های زیر به‌موقع پیش می‌روند؟  
    - DB Design & Migration Scripts  
    - Plugin Skeleton (`corlink-core`)  
    - REST API Contract و Auth  
    - Caching & Performance Strategy  
    - CI/CD Pipeline و Docker/Local Setup  

### پنج‌شنبه
- دریافت **Architecture Review Summary**:
  - Diagramها، ERD، لیست Endpoints  
  - ریسک‌های فنی باقی‌مانده  

### جمعه
- تصمیم رسمی:
  - Architecture v1 برای MVP آماده است  
  - اجازه شروع Sprint 1 (Auth & Infra) طبق فاز ۴

## فاز ۴ – توسعه MVP (Sprint 0 تا 10)

در این بخش، فقط زاویه **مدیریت و Cross-team** را برای هر Sprint می‌دهیم؛ جزئیات روزانه فنی در `Phase4-Daily-Execution-Plan.md` آمده است.

### Sprint 0 – آماده‌سازی فنی (هفته ۱ از فاز ۴)

- **شنبه:** تأیید لیست کارهای Setup (Dev/Staging/Git/CI)  
- **سه‌شنبه:** چک سلامت محیط‌ها (آیا تیم می‌تواند بدون مشکل کدنویسی و تست کند؟)  
- **پنج‌شنبه:** دریافت گزارش راه‌اندازی (چه مشکلاتی رخ داد؟)  
- **جمعه:** تأیید آغاز Sprint 1  

### Sprint 1 – Auth & پایه زیرساخت (هفته ۲–۳ فاز ۴)

- **شنبه (Planning):** حضور مدیریت/محصول برای تأکید روی اهمیت تجربه ثبت‌نام ساده و امن  
- **سه‌شنبه:** بررسی ریسک‌:
  - مشکلات Email Delivery  
  - UX پیچیده در ثبت‌نام  
- **پنج‌شنبه:** Demo Auth Flow، تصمیم روی تغییرات سریع قبل از رفتن به Profile  
- **جمعه:** تأیید این‌که Auth برای MVP پذیرفتنی است (هرچند ممکن است UI هنوز کامل نباشد)

### Sprint 2 و 3 – Profile System (هفته ۴–۷ فاز ۴)

- **مدیریت باید دنبال کند:**
  - آیا «Profile Completion Rate» در تست‌های داخلی خوب است؟  
  - آیا فرم‌ها خیلی طولانی/پیچیده نیستند؟  
- **پنج‌شنبه هر هفته:** Demo تکمیل پروفایل واقعی انتهابه‌انتها  
- **جمعه‌ها:**  
  - تصمیم روی Scope تنظیمات حریم خصوصی (چه‌چیزی برای MVP ضروری است، چه‌چیزی خیر)  

### Sprint 4 و 5 – Matching & Search (هفته ۸–۱۱ فاز ۴)

- **Focus مدیریتی:**
  - هسته ارزش پلتفرم همین‌جاست؛ نیاز به توجه ویژه  
- **شنبه‌ها:**  
  - تاکید بر KPIها: Match Success Rate, User Satisfaction با پیشنهادات  
- **سه‌شنبه‌ها:**  
  - درخواست گزارش کوتاه از تیم درباره «کیفیت پیشنهادات» در تست‌های داخلی  
- **پنج‌شنبه‌ها:**  
  - Demo صفحه Suggestions + Search + توضیح Match Score  
- **جمعه:**  
  - تصمیم روی این‌که آیا برای MVP، کیفیت الگوریتم در حد قابل‌قبول هست یا نیاز به Sprint اضافی دارد  

### Sprint 6 و 7 – Messaging & Requests (هفته ۱۲–۱۵ فاز ۴)

- **مدیریت:**  
  - نظارت بر سادگی جریان ارتباط (Send Message/Request → Accept → Start conversation)  
  - اطمینان از این‌که محدودیت‌ها/Throttleها کاربران را ناراضی نکنند  
- **پنج‌شنبه‌ها:**  
  - Demo استخدامی واقعی بین دو اکانت تست (از Discovery تا شروع گفتگو/همکاری)  

### Sprint 8 – Dashboard & Landing (هفته ۱۶–۱۷ فاز ۴)

- **Focus مدیریتی:**  
  - آیا Landing Page پیام واضحی دارد؟  
  - آیا Dashboard برای کاربر جدید گیج‌کننده نیست؟  
- **پنج‌شنبه:** Demo Landing و Dashboard  
- **جمعه:** تصمیم روی پیام‌های اصلی مارکتینگ (هم‌راستا با فاز ۵ – Go-to-Market)  

### Sprint 9 – Optimization & Polish (هفته ۱۸–۱۹ فاز ۴)

- **مدیریت باید مطمئن شود:**
  - Performance, Security, SEO در سطح MVP قابل‌قبول هستند.  
  - چک‌لیست‌های تکنیکی در Phase4 و Phase3 برای این بخش تیک می‌خورند.  
- **پنج‌شنبه:** گزارش Performance (PageSpeed, LCP, CLS,…) + Security Audit خلاصه  

### Sprint 10 – Beta Test & Launch Preparation (هفته ۲۰–۲۱ فاز ۴)

- **شنبه–سه‌شنبه:**  
  - نظارت روی جذب Beta Users (هم‌راستا با فاز ۵ – Pre-Launch)  
  - جمع‌آوری و مرور سریع بازخوردها  
- **پنج‌شنبه:**  
  - تصمیم روی:  
    - آیا MVP آماده Beta عمومی است؟  
    - آیا باگ‌های Critical رفع شده‌اند؟  
- **جمعه:**  
  - هماهنگی نهایی با تیم Legal/Marketing (فاز ۵) برای روز Launch

## از دید مدیریت: چه چیزهایی هر روز باید چک شود؟

### شنبه تا چهارشنبه
- **مدیریت/PM:**
  - حضور در Daily Standup (یا دریافت خلاصه آن) برای:
    - دیدن Blockerها  
    - اطمینان از تمرکز تیم روی اولویت‌های درست  
  - چک برد/ابزار مدیریت پروژه (Jira/Notion/Trello):
    - Stories مهم این هفته در حال پیشرفت هستند؟  

### پنج‌شنبه
- تهیه و مرور ۳ نوع گزارش:
  - **Product Report:** پیشرفت Features نسبت به Roadmap  
  - **Tech Report:** ریسک‌های فنی، مشکلات Performance/Security  
  - **UX/Users Report:** بازخورد از تست‌ها/Beta (اگر در جریان است)  

### جمعه
- تبدیل گزارش‌ها به **تصمیمات عملی**:
  - تغییر یا تثبیت Scope  
  - جابه‌جایی Featureها (قبل/بعد از MVP)  
  - اضافه‌کردن Sprint Buffer در صورت نیاز
